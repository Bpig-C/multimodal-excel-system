# 上传/导出中断问题说明与初步方案

## 问题描述

当前“Excel 上传”“图片 ZIP 上传”“批量导出 ZIP”虽然在前端表现为异步请求，但本质上仍然是“单个 HTTP 请求内完成全部工作”的模式：

- 上传接口：请求发出后，服务端在同一个请求内完成文件接收、保存、解析、入处理结果。
- 批量导出接口：请求发出后，服务端在同一个请求内完成数据读取、格式转换、ZIP 打包，并直接返回文件流。

这意味着：

- 站内切换路由时，请求通常不会立刻中断，但页面上的进度 UI 会丢失。
- 刷新页面、关闭标签页、跳转外部站点时，请求大概率会被浏览器中断。
- 导出不属于后台任务，用户离开页面后无法继续查看状态，也无法在任务完成后再回来领取结果。

## 当前影响

- 大文件上传或导出时，用户对“是否还在执行”缺少稳定预期。
- 任务执行过程与页面生命周期耦合较深，交互韧性不足。
- 后续如果导出数据量继续增大，超时、中断、重复发起的概率会明显升高。

## 初步方案

建议后续把“长耗时上传/导出”逐步改成“后台任务 + 前端查询状态”的模式。

### 方案目标

- 用户切换站内页面后，任务仍可继续执行。
- 页面刷新后，仍可根据任务 ID 恢复进度展示。
- 导出完成后，用户可以单独下载结果文件，而不是依赖原始请求一直保持连接。

### 推荐改造顺序

1. 优先改造“批量导出”
- 收益最大，因为导出最适合后台任务模型。
- 前端先提交导出请求，后端立即返回 `job_id`。
- 后端异步执行数据收集、转换、ZIP 打包。
- 前端通过轮询或 WebSocket 查看状态：`pending/running/completed/failed`。
- 完成后调用单独下载接口获取文件。

2. 再评估“Excel 上传”
- 如果上传文件本身不大，但“解析/入库”耗时长，可拆成两阶段：
- 第一阶段只负责文件上传。
- 第二阶段基于已上传文件创建后台处理任务。

3. 图片 ZIP 上传可先保持现状
- 如果只是纯上传且文件通常不大，短期可以不做后台任务化。
- 若后续图片包明显增大，再复用同一套任务框架。

## 最小可落地设计

### 后端

- 增加任务表，例如记录：
- `job_id`
- `job_type`
- `status`
- `progress`
- `message`
- `result_path`
- `error_message`
- `created_at`
- `updated_at`

- 新增接口：
- `POST /api/v1/export/corpus/{processor}/batch/jobs`：创建导出任务
- `GET /api/v1/jobs/{job_id}`：查询任务状态
- `GET /api/v1/jobs/{job_id}/download`：下载结果文件

- 后台执行可先用最轻量方式落地：
- 先用应用内后台线程/任务池
- 如果后续任务量上来，再迁移到 Celery、RQ 或独立 worker

### 前端

- 发起任务后保存 `job_id`
- 在页面内轮询状态
- 将任务状态存入 store，支持切页后继续查看
- 任务完成后展示“下载结果”按钮，而不是依赖原始请求直接回包

## 风险与注意点

- 应避免结果文件无限堆积，需要增加过期清理策略。
- 如果采用应用内后台线程，服务重启后任务恢复能力较弱。
- 若未来需要更高稳定性，建议预留迁移到独立 worker 的空间。

## 建议结论

短期内最值得做的是“批量导出后台任务化”。  
它能最直接解决“切页后感知丢失、刷新后任务中断、长连接不稳定”的问题，同时对现有上传链路影响最小。
